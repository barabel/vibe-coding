# Глобальные инструкции для Claude Code

## Контекст проекта

- перед каждым запросом читать `CONTEXT.md` в корне проекта, если существует
- перед каждым запросом читать `UBIQUITOUS_LANGUAGE.md` в корне проекта, если существует

## Общее

- во всех ответах и commit-сообщениях — максимальная краткость, грамматикой можно жертвовать ради лаконичности

## Планирование

- в конце каждого плана — список нерешённых вопросов (если есть), максимально кратко, грамматикой можно жертвовать

## Стиль кода

- каждый атрибут JSX-элемента — на отдельной строке
- в колбэках (.map, .reduce и т.д.) использовать тело функции `{}` с `return`, деструктурировать нужные поля до `return`
- не писать inline: если выражение не помещается в одну строку — разворачивать

## Языки

- отвечать только на русском. Технические термины и идентификаторы кода — в оригинале

## Принципы разработки

### 1. Думай перед кодом

**Не предполагай молча. Выявляй трейдоффы.**

Перед реализацией:
- явно называй допущения; если не уверен — спроси
- если возможны несколько трактовок — перечисли, не выбирай молча
- если есть более простой подход — скажи, предложи
- если что-то непонятно — останови, назови что именно, спроси

### 2. Простота прежде всего

**Минимум кода, решающего задачу. Ничего спекулятивного.**

- никаких фич сверх запрошенного
- никаких абстракций для единственного использования
- никакой «гибкости» или «конфигурируемости» без запроса
- никакой обработки ошибок для невозможных сценариев
- если написал 200 строк, а можно 50 — перепиши

Спроси себя: «Скажет ли senior-инженер, что это overcomplicated?» Если да — упрости.

### 3. Хирургические изменения

**Трогай только то, что необходимо. Убирай только свой мусор.**

При редактировании существующего кода:
- не «улучшай» соседний код, комментарии, форматирование
- не рефакторь то, что не сломано
- соответствуй существующему стилю, даже если ты бы сделал иначе
- заметил мёртвый код — упомяни, не удаляй

После своих изменений:
- удаляй импорты/переменные/функции, осиротевшие из-за твоих правок
- не трогай pre-existing мёртвый код без запроса

Критерий: каждая изменённая строка должна прямо следовать из запроса пользователя.

### 4. Исполнение с критерием успеха

**Определяй критерий успеха. Итерируй до достижения.**

Преобразуй задачи в проверяемые цели:
- «Добавь валидацию» → «Напиши тесты для невалидных входных данных, затем сделай так, чтобы они проходили»
- «Исправь баг» → «Напиши тест, воспроизводящий его, затем сделай так, чтобы он проходил»
- «Рефактор X» → «Убедись, что тесты проходят до и после»

Для многошаговых задач — краткий план:
```
1. [Шаг] → проверка: [что проверяем]
2. [Шаг] → проверка: [что проверяем]
3. [Шаг] → проверка: [что проверяем]
```

Чёткий критерий успеха позволяет работать независимо. Размытый («сделай так, чтобы работало») требует постоянных уточнений.
