Используй этот гайд, чтобы подсказать подходящий пайплайн:

---

## Все скиллы

| Скилл | Что делает | Когда использовать |
|---|---|---|
| `/zoom-out` | Карта модулей и вызывающего кода вокруг компонента | Не знаешь кодбейс или как компонент вписывается в систему |
| `/domain-model` | Интервью по доменной модели, шарпит термины, обновляет `CONTEXT.md` | Онбординг, перед большими изменениями |
| `/ubiquitous-language` | Формализует термины из разговора в `UBIQUITOUS_LANGUAGE.md` | После `/domain-model`, когда нужен глоссарий |
| `/grill-me` | Стресс-тест плана — задаёт вопросы по всем развилкам | Есть идея фичи, но план нечёткий |
| `/triage-issue` | Исследует кодбейс, находит root cause, пишет TDD-план в `docs/issues/` | Нужно зафиксировать и распланировать баг |
| `/to-prd` | Синтезирует PRD из текущего контекста разговора → `docs/prd/` | Наговорил фичу в чате, хочешь зафиксировать |
| `/to-issues` | Разбивает план/PRD на независимые тикеты → `docs/issues/` | После `/to-prd`, или сложный баг на несколько компонентов |
| `/design-an-interface` | Генерирует 3+ радикально разных дизайна интерфейса модуля | Проектируешь новый модуль, непонятно как его оформить |
| `/request-refactor-plan` | Интервью + план tiny commits → `docs/issues/` | Рефактор компонента |
| `/tdd` | RED→GREEN→REFACTOR циклами по плану | Реализация — после любого планировочного скилла |
| `/improve-codebase-architecture` | Ищет архитектурные проблемы, предлагает кандидатов на рефактор | Онбординг, или не знаешь с чего начать рефактор |

## Пайплайны

### 0. Онбординг на незнакомый / старый проект
```
/domain-model → /ubiquitous-language
```
- `/domain-model` — пиши: _"Я делаю онбординг на этот проект. Исследуй кодбейс и давай разберём термины предметной области."_
- `/ubiquitous-language` — формализует все термины из разговора

> Если нужна архитектурная карта а не только термины — замени `/domain-model` на `/improve-codebase-architecture`.

### 1. Фикс бага
```
/triage-issue → (опц: /to-issues) → /tdd
```
- `/triage-issue` — описываешь баг, находит root cause, пишет TDD-план
- `/to-issues` — только если баг сложный и затрагивает несколько компонентов
- `/tdd` — реализует по RED→GREEN циклам

### 2. Новая фича
```
[описываешь фичу] → /grill-me → /to-prd → /to-issues → /tdd
```
- `/grill-me` — стресс-тест плана (опционально, если план нечёткий)
- `/to-prd` — синтезирует PRD → `docs/prd/<slug>.md`
- `/to-issues` — разбивает на вертикальные слайсы
- `/tdd` — реализует тикет за тикетом

> Если нужен новый модуль — перед `/tdd` вставляй `/design-an-interface`.

### 3. Изменение функциональности компонента
```
/zoom-out → [описываешь изменение] → /to-prd → /to-issues → /tdd
```
- `/zoom-out` — карта вызывающего кода (опционально)
- Дальше — тот же флоу что у фичи

### 4. Рефактор компонента
```
/zoom-out → /request-refactor-plan → /tdd
```
- `/zoom-out` — контекст компонента (опционально)
- `/request-refactor-plan` — интервью, план tiny commits
- `/tdd` — реализует шаг за шагом

> Если не знаешь что рефакторить — `/improve-codebase-architecture` найдёт кандидатов.

---

На основе ситуации пользователя: выбери 1-2 наиболее подходящих пайплайна, выдели **нужный пайплайн жирным**, и кратко объясни почему именно он (1-2 предложения). Если ситуация неоднозначная — уточни один вопрос.
